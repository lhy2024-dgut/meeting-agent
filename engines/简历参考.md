  项目简介（一句话）

  面向隐私敏感场景（科研团队、法务、医疗）的全本地化 AI 会议纪要系统，基于 LangChain + Ollama 本地大模型 + PGVector向量库，覆盖语音转录→场景化纪要生成→跨会议语义检索→多轮对话问答完整链路，数据不出本地。

  ---
  项目亮点

  ① ASR 转写引擎（负责人）

  使用 faster-whisper 与 FunASR SenseVoiceSmall 双引擎框架实现语音转写，包括静音检测切块、并行转写、时间戳对齐模块。通过
  ffmpeg silencedetect 在语音停顿处切割音频，结合 ProcessPoolExecutor 4 进程并行转写，将 6 分钟录音的处理耗时从约 10
  分钟压缩至 50 秒以内。支持前端单选模型切换功能，保证两套引擎输出统一的 {id, text, start, end} segment
  格式，解决多模型接口不兼容问题。设计 per-meeting 术语词表系统，通过 initial_prompt / hotword 参数将用户自定义术语注入
  ASR 引导识别，使用 token 估算与自动截断防止超出 200 token 上下文限制。

  ② 场景化纪要生成链（负责人）

  使用 LangChain LCEL 管道从零实现结构化纪要生成链，包括 Prompt 模板加载、ChatOllama 调用、StrOutputParser
  输出解析模块。通过三级容错机制（JSON 直接解析 → 修复裸换行符再解析 → 旧式三段式正则回退）保证本地小模型 JSON
  输出可用性，将输出成功率从约 70% 提升至 99%。支持 4
  种场景化模板（学术组会、客户访谈、立项评审、项目周会）以及用户自定义标题功能，通过 YAML
  动态加载模板保证扩展无需改代码，解决通用提示词对垂直场景结构适配差的问题。

  ③ 向量检索与 RAG（参与）

  基于 PostgreSQL + PGVector 扩展构建向量存储，设计 ivfflat 余弦索引（lists=100）与覆盖式索引重建策略，结合 BM25
  实现混合检索，引入 BGE Reranker 对召回结果重排序。基于 AliMeeting 数据集（含 118
  小时真实中文会议录音）构建评估体系，测得 Recall@5 = 96.15%，MRR = 0.8923；量化分析 ASR 错误对 RAG
  检索召回率的影响（ASR 输入 vs 人工转录对照），使用 pgvector psycopg2 适配器注册机制防止跨连接类型转换失败。

  ④ 多轮对话 Agent（参与）

  构建基于 LangGraph StateGraph 的会议问答 Agent，使用 MemorySaver 实现 10 轮滑动窗口对话记忆，按 meeting_id
  隔离各会议上下文，优化长对话场景下 Token 占用，支持追问问答与跨会议语义检索联动。

  面试常见追问参考答案

  Q：为什么选本地部署而不用 OpenAI API？

  ▎ 项目定位面向科研机构、法务、医疗等数据敏感场景，录音内容涉及未发表实验数据或商业机密，云端传输不可接受。本地化还带来
  ▎ 另一个优点：离线可用、零 API 费用，降低了实际推广门槛。

  Q：faster-whisper 和 SenseVoiceSmall 怎么选型？

  ▎ 从 issue #18 的调研报告看：SenseVoiceSmall 的推理速度是 Whisper-Large-v3 的 15 倍，适合 CPU
  ▎ 环境快速处理；faster-whisper 通过 initial_prompt
  ▎ 支持更灵活的术语注入、对中文长尾词的可控性更强。因此两者并存，由用户在前端选择，方便自行做 CER 对比评测。

  Q：并行 ASR 如何保证切割点不截断词语？

  ▎ 用 ffmpeg silencedetect 滤镜检测 ≥500ms
  ▎ 的静音段，取静音段中点作为切割点，保证切割一定落在说话停顿处。每块独立转写后用 start_offset
  ▎ 修正时间戳，再按时间排序合并，避免乱序。

  Q：PGVector 相比 Chroma/Milvus 有什么取舍？

  ▎ 项目本身已用 PostgreSQL 管理会议元数据、转录文本，PGVector
  ▎ 让向量与关系数据共享同一个数据库，省去双数据库一致性维护、双套备份的成本，事务和权限管理全部复用 Postgres 生态。对于
  ▎ ≤万级 chunk 的场景性能完全满足，且部署更简单（CREATE EXTENSION vector 即可）。
