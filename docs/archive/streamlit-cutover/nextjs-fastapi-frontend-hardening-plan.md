# Next.js + FastAPI 前端补强计划报告

## 1. 背景与目的

本报告面向 `Meeting Agent` 当前迁移中的新前端实现：

- 前端：`C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\web`
- 后端：`C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\api`
- 旧版入口：`C:\Users\Administrator\Desktop\meeting-agent\app.py`

当前迁移版前端已经覆盖核心链路，包括：首页、历史页、上传页、会议详情页、统计页、单场会议问答、跨会议问答，以及导出和重新生成纪要等能力。

现阶段的主要矛盾已经不是“页面能不能跑”，而是：

1. 能否在并行观察阶段保持稳定。
2. 能否把长任务、聊天、错误处理这些高复杂度链路收敛成可维护形态。
3. 能否为后续默认入口切换提供足够稳的工程基础。

本报告的目标是把“前端最该补强的 5 个点”整理成一份可执行计划，供排期、分工和验收使用。

## 2. 当前状态摘要

根据现有代码和验收文档，迁移版前端具备以下特点：

- 使用 `Next.js 16 + React 19 + TypeScript`，页面和组件目录已成型。
- 与 `FastAPI` 配合，已经形成前后端分离形态。
- 已接入 `Playwright`，覆盖主链路点击验收。
- 仍以手写 `fetch`、页面局部状态和客户端组件为主。
- 长任务、聊天、错误处理、缓存和可观测性仍偏 MVP 实现。

结论：
当前版本已经适合并行观察，但还不适合在缺少补强的情况下直接成为默认入口。

## 3. 总体补强原则

本轮补强建议遵循以下原则：

1. 先稳主链路，再做交互优化。
2. 先收敛状态模型，再扩展新能力。
3. 先解决长任务与聊天的不确定性，再做 SSR/性能优化。
4. 优先补可观测性和错误承接，降低联调与验收成本。
5. 不重写现有前端，基于当前目录做增量演进。

## 4. 五个最高优先级补强点

### 4.1 优先级 1：统一前端数据层与请求基础设施

#### 现状

当前请求主要集中在：
`C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\web\src\lib\api.ts`

存在的问题：

- 请求封装过薄，只有最基础的 `fetch` 包装。
- 没有统一的超时、重试、取消、退避、错误解析策略。
- 页面各自管理 loading/error/success 状态，逻辑分散。
- `router.refresh()`、局部 state 和手工刷新混用，后续容易失控。

#### 目标

建立统一的数据访问层，使会议列表、详情、任务、聊天会话等能力具备一致的请求行为和状态管理方式。

#### 建议动作

- 引入统一请求适配层，补齐：超时、Abort、结构化错误解析、基础日志。
- 新增页面级 hooks：`useMeetings`、`useMeetingDetail`、`useJobStatus`、`useChatSession`。
- 统一定义请求状态：`idle / loading / success / error / retrying`。
- 对写操作统一设计刷新与失效策略，减少页面内手动刷新。
- 评估是否引入 `TanStack Query`；若当前不引入，也应先形成自有 hooks 层。

#### 预期收益

- 降低页面组件复杂度。
- 降低重复请求逻辑和刷新逻辑。
- 为任务轮询、聊天恢复、错误处理提供统一底座。

### 4.2 优先级 2：补强长任务状态模型与恢复能力

#### 现状

当前上传和重新生成主要依赖：

- 创建任务后前端轮询 `GET /api/jobs/{job_id}`
- 页面本地持有当前任务状态
- 刷新后任务上下文丢失

上传页实现集中在：
`C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\web\src\components\upload\upload-page.tsx`

存在的问题：

- 任务轮询逻辑写在页面内，无法复用。
- 页面刷新、关闭、跳转后，用户无法恢复当前任务。
- 没有任务历史视图或“最近任务”承接入口。
- 没有轮询退避、连续失败处理和重连策略。

#### 目标

将“会议处理任务”提升为一等前端对象，使用户在上传、重新生成等长任务场景中有稳定、可恢复的体验。

#### 建议动作

- 抽出统一 `useJobPolling(jobId)` 或等价能力。
- 在 URL、sessionStorage 或 localStorage 中保留最近任务上下文。
- 为上传页和重新生成页统一任务展示组件。
- 设计任务中断/失败恢复文案和操作入口。
- 二期预留 SSE 替代轮询的接入点。

#### 预期收益

- 上传与重新生成体验更稳。
- 长任务不再强依赖页面存活。
- 更接近真实产品，而不是演示态交互。

### 4.3 优先级 3：补强聊天系统的正式化能力

#### 现状

聊天系统当前已实现：

- 单会议问答
- 跨会议问答
- 来源展示
- 来源跳转与高亮
- 本地缓存辅助来源预览

相关代码主要在：

- `C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\web\src\components\chat\chat-workspace.tsx`
- `C:\Users\Administrator\Desktop\meeting-agent\migration\nextjs-fastapi\web\src\components\meeting\meeting-detail-page.tsx`

存在的问题：

- 会话刷新后不可恢复。
- 没有流式响应，长回答感知较弱。
- 缺少“停止生成”“重新发送”“消息级失败重试”等能力。
- 会议上下文缓存使用模块级 `Map`，适合当前版本，但长期维护风险高。
- 错误承接和会话生命周期管理还比较薄。

#### 目标

把聊天从“可用功能”提升为“可稳定使用的产品能力”。

#### 建议动作

- 为聊天会话引入前端会话恢复能力，至少保留 `session_id`。
- 支持流式响应或为流式响应保留接口边界。
- 增加取消发送、重试发送、消息失败重发入口。
- 将来源预取和缓存逻辑下沉为独立 hook / service。
- 统一聊天状态模型：`bootstrapping / ready / sending / streaming / failed`。

#### 预期收益

- 聊天体验从演示态走向产品态。
- 降低跨会议问答复杂逻辑在组件内部继续膨胀的风险。

### 4.4 优先级 4：建立统一错误处理与可观测性能力

#### 现状

当前前端错误承接主要是抛 `Error` 并显示字符串消息，典型问题包括：

- 看不到结构化业务错误。
- 很难区分 API 错误、网络错误、任务失败、后端推理失败。
- 页面上的错误提示可读性不足，联调时定位成本高。

已有相关问题梳理文档：
`C:\Users\Administrator\Desktop\meeting-agent\docs\frontend-migration-issue-report.md`

#### 目标

建立统一的前端错误模型和最小可观测性能力，降低联调、验收、灰度观察成本。

#### 建议动作

- 统一解析 API 错误结构：`code / message / details`。
- 请求层补超时与错误类型映射。
- 页面层统一空态、失败态、重试态组件。
- 对长任务、聊天、导出等关键操作加上最小日志埋点。
- 在开发环境补健康检查和 API 连接状态提示。

#### 预期收益

- 明显降低联调排查成本。
- 增强并行观察阶段的问题定位效率。
- 为后续默认入口切换提供稳定支撑。

### 4.5 优先级 5：收敛客户端组件边界，逐步发挥 Next.js 优势

#### 现状

当前很多核心页面组件使用了 `"use client"`，包括上传、历史、聊天、统计和详情等。
这在迁移早期是合理的，但也说明：

- 页面级组件承担了较多副作用和状态逻辑。
- 只读内容和高交互内容尚未充分拆分。
- 当前更多是在“用 React 写页面”，而不是充分利用 `Next.js` 的服务端边界。

#### 目标

逐步明确 SSR/CSR 边界，减少高层客户端组件的职责，保留 Next.js 在只读页面、结构化展示和数据边界上的优势。

#### 建议动作

- 将详情页、历史页中的只读数据展示部分尽量保持服务端数据获取。
- 客户端组件只保留搜索输入、任务状态、聊天、跳转高亮等强交互部分。
- 拆分页面级大组件，减少“请求 + 状态 + 展示 + 副作用”耦合在同一文件中的情况。
- 对图表和 Markdown 区块评估是否需要更清晰的 lazy/load 边界。

#### 预期收益

- 组件职责更清晰。
- 后续功能迭代更可控。
- 为性能和维护性打基础。

## 5. 分阶段执行计划

### 阶段 A：基础设施收敛（P0）

目标：先补底座，避免后续每加一个功能都继续复制薄封装模式。

任务：

1. 重构 `src/lib/api.ts`，加入统一错误解析、超时、取消。
2. 新增基础 hooks：`useApiRequest`、`useJobPolling`、`useChatSession` 或等价抽象。
3. 统一错误提示组件和加载状态组件。
4. 为开发环境增加 API 健康提示或连接失败提示。

验收标准：

- 会议列表、详情、任务、聊天至少各有一处改为统一数据层调用。
- 页面不再直接散落重复的 `try/catch + setError + setLoading` 逻辑。

### 阶段 B：长任务与聊天补强（P0）

目标：解决最影响用户体验和系统稳定性的两条复杂链路。

任务：

1. 上传页任务轮询逻辑抽离并支持任务恢复。
2. 重新生成纪要接入同一任务状态模型。
3. 聊天会话保留 `session_id`，支持刷新后恢复。
4. 增加聊天失败重试与清晰错误提示。
5. 为流式聊天预留接口边界。

验收标准：

- 上传任务刷新页面后仍可恢复查看。
- 重新生成任务与上传任务体验一致。
- 聊天页面刷新后可恢复会话或明确恢复失败原因。

### 阶段 C：页面结构与组件边界收敛（P1）

目标：降低客户端大组件复杂度，提升后续维护性。

任务：

1. 拆分 `history-page.tsx`、`chat-workspace.tsx`、`meeting-detail-page.tsx` 等大组件。
2. 将只读展示部分与高交互部分拆开。
3. 收敛缓存和副作用逻辑到 hooks 或 service。
4. 补充页面级状态图和边界说明。

验收标准：

- 页面大组件职责明显收敛。
- 缓存、轮询、跳转定位等逻辑不再散落在页面主体中。

### 阶段 D：验收与切换准备（P1）

目标：为后续默认入口切换做准备，而不是停留在“人工感觉可以”。

任务：

1. 扩展 Playwright 用例，覆盖异常态和恢复态。
2. 增加 API 不可用、任务失败、聊天失败等回归场景。
3. 固化并行验收脚本与观察清单。
4. 输出切换门槛与回退方案。

验收标准：

- smoke 和 full E2E 稳定通过。
- 异常链路有自动化覆盖。
- 默认入口切换具备明确前置条件。

## 6. 建议排期

若按单人或小团队持续推进，建议如下：

- 第 1 周：阶段 A
- 第 2 周：阶段 B
- 第 3 周：阶段 C
- 第 4 周：阶段 D 与并行观察补差

若资源有限，建议至少先完成：

1. 请求层与错误模型统一。
2. 长任务恢复。
3. 聊天会话恢复与失败重试。

这三个点的收益最高。

## 7. 风险与依赖

### 风险 1：前端补强过程中牵动后端 API 形态

说明：
统一错误结构、任务恢复和聊天恢复，可能需要 API 返回更清晰的状态字段。

应对：
前后端联合定义接口契约，优先保证兼容新增，而不是直接破坏现有接口。

### 风险 2：客户端状态改造期间出现回归

说明：
当前页面虽不完美，但主链路已能跑通。重构数据层和组件边界时容易引入回归。

应对：
每个阶段都必须配套回归测试和 Playwright 冒烟验证。

### 风险 3：过早追求 Next.js 理想形态，导致补强周期过长

说明：
当前最重要的是稳定性和工程收敛，不是立刻把所有页面改成最优 SSR/CSR 结构。

应对：
坚持“先稳主链路，再做结构优化”的顺序。

## 8. 切换建议

在以下条件满足前，不建议下线旧版 Streamlit 主入口：

1. 请求层与错误模型已经统一，不再依赖页面各自兜底。
2. 上传和重新生成任务支持恢复查看。
3. 聊天会话具备基本恢复或明确失败承接能力。
4. 关键异常链路已有自动化覆盖。
5. 连续一段并行观察期内，没有出现阻断性问题。

## 9. 最终结论

当前迁移版前端已经完成“功能主链路迁移”这个阶段，接下来最该做的不是继续堆页面，而是进入“工程补强阶段”。

优先顺序应明确为：

1. 统一前端数据层与请求基础设施。
2. 补强长任务恢复能力。
3. 补强聊天的正式化能力。
4. 建立统一错误处理与可观测性。
5. 逐步收敛客户端组件边界。

只要这五个方面按顺序落地，新前端就更有条件从“迁移版”走向“默认入口”。
