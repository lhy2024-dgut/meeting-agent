# Meeting Agent 迁移开发任务清单 + 优先级 + 工期

## 1. 使用说明

本清单基于 [nextjs-fastapi-migration-plan.md](C:/Users/Administrator/Desktop/meeting-agent/docs/nextjs-fastapi-migration-plan.md) 拆解，目标是把迁移方案转换成可执行排期。

适用范围：
- 单人开发或 2 人小组开发
- 本地部署优先
- 先完成可用版本，再考虑增强项

优先级定义：
- `P0`：主链路必做，不完成无法上线替代 Streamlit
- `P1`：强烈建议做，缺失会明显影响可用性
- `P2`：增强项，可放到第二阶段

工期说明：
- 以下工期按“已有当前仓库上下文、单人开发、每天有效开发 6-7 小时”估算
- 单位默认用“人天”
- 不含大范围返工时间

---

## 2. 总体排期建议

### 2.1 MVP 目标

MVP 版本应至少覆盖：
- 首页
- 历史页
- 详情页
- 统计页
- 上传处理
- 导出
- 单会议聊天

跨会议聊天、流式聊天、持久化会话可放在第二阶段。

### 2.2 建议总工期

| 版本 | 范围 | 预估工期 |
|---|---|---|
| MVP | 替代 Streamlit 核心链路 | 18-23 人天 |
| V1 | MVP + 跨会议聊天 + 更稳的任务与会话模型 | 24-30 人天 |

### 2.3 关键路径

关键路径按顺序是：
1. API 骨架
2. 只读 API
3. Next.js 前端骨架
4. 低风险页面
5. 任务系统
6. 上传页
7. 导出与重新生成
8. 聊天
9. 联调与切换

## 10. 风险缓冲建议

建议额外预留：
- `2-3 人天` 作为联调和返工缓冲

最容易超时的部分：
1. 上传任务状态与异常处理

3. 聊天会话状态
4. 前后端字段命名不一致

---

## 11. 最终建议

如果你想稳一点，就按 `M1 -> M2 -> M3` 推进。
如果你想快一点，就先把聊天范围收缩到“详情页内单会议聊天”，不要一开始就做跨会议模式、流式返回、持久化会话。

最现实的路线是：
1. 先做读能力
2. 再做上传任务
3. 再做导出
4. 最后做聊天

这样最符合你当前仓库的结构，也最不容易在中途卡住。
